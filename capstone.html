What I Learned from my Capstone

Emma Livingston

5/23/2020


One of the required classes for the Statistical and Data Sciences major at Smith College is the Capstone. Each group of 3-5 students works with a client in either non-profit, industry, government, or academia, to complete a data-related project.

I learned a lot of new coding skills through this project, but what I want to focus on here is what I learned outside of the end result of the project. Mostly, this is a (slightly snarky) note-to-my-past-self, and what I wish I had thought about starting in January. C‚Äôest la vie, better late than never.


Working with Senstive Data

Our team was tasked with working with data that could not be shared publicly. This conflicted with our typical workflow in the department of using GitHub. After much consideration, we decided to use a private repository for the project, but without uploading any of the data. This took some planning ahead.

We each had a data folder in our .Rproj, but this folder was listed on the .gitignore, so we didn‚Äôt commit or push any of the changes we made to that folder. This meant that the data wasn‚Äôt uploaded to GitHub, but it also meant we had to be intentional about making sure we were all on the same page with the contents of the data folder.

The way we worked around this was by using the here package, and making sure that we all had the same exact .csv files with the same names in our data folder. Then, when we wanted to import data in our repository, the code looked like this:

df <- read_csv(here("/data/frame.csv"))


Organizing the Code

Something I wish we had done better about from the beginning was how we organized our .Rmds. Based on our experience and what I wish I would have thought of in February, here are some tips for keeping a project organized.

  1. Pick one document to do all the data cleaning in. If you have the same data cleaning process copied and pasted into all of the documents you need, there are bound to be inconsistencies, which will lead to some late night nit-picking of inner joins (hypothetically). This way, you can just run that script before any of your other stuff. Alternatively, if you have a better way to go about this, I‚Äôd love to hear your ideas.

  2. Make sure that the name of the document tells you exactly what is inside. Don‚Äôt just tack on something to the end of a document you are already in, start a new one. Leaving your ‚Äúcase study‚Äù code at the bottom of your exploratory_data_analysis.Rmd doesn‚Äôt help anyone. Plus, the more that work is spread out across documents, the fewer merge conflicts you‚Äôre going to run into.

  3. Comment your code. Not for yourself, but for the other people who have to read it. While it might be self-explanatory and very clear to you what you are doing, it might not be so clear to someone else! You may end up in a situation where someone else has to pick up where you left off, so be kind to them and make sure it is very clear what your goal is and how you got there.

  4. Commit and push early and often. GitHub is only a tool that is as valuable as how well you use it, so do future-you a favor and make sure your work is pushed properly.

  5. Don‚Äôt just work in the console! If you are trying to figure out a problem and you‚Äôre playing around in the console, consider saving more of it than you might think. The stuff you‚Äôre playing around with right now might really help you out later! And if it doesn‚Äôt, you can just comment it out. Nobody cares about the thing you are (probably) most proud of

As a data scientist, what you are probably most proud of in the project are the new tech skills you developed, the really intricate visualization that you figured out how to do, or how you finally figured out how to get the data in the format you needed.
